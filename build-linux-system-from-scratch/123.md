# OSViz 设计文档

## 系统目标

基于 **2025 年全国大学生 OS 比赛现场赛测例**（`testsuits-for-oskernel-on-site-final-2025`），在自研操作系统内核上搭建 observability 基础设施，支撑现场赛核心验证：

> 在 QEMU RISC-V64 / LoongArch64（virtio-net / virtio-block）及对应物理开发板环境下，验证内核能够访问文件系统并**正确执行 Linux 软件**。

### 测例覆盖的用户程序

与官方 Alpine ext4fs 镜像保持一致（版本见 `soft-info.txt`）：

| 程序 | 版本（参考） | 现场赛用途 |
|------|-------------|-----------|
| `git` | 2.47.3 | init / add / commit / log / clone / push / pull |
| `vim` | 9.1 | 加载运行、编辑保存 `hello.c` |
| `gcc` | 14.2.0 | 编译运行 `hello.c` → `Hello, World!` |
| `rustc` | 1.83.0 | 编译运行 `helloworld.rs` → `Hello, World!` |
| `busybox` / `bash` | — | shell 与基础工具链 |

### OSViz 三大能力

1. **日志记录**（事件流 + 按需快照）
2. **日志生成**（测例程序写事件；Linux 命令按需采快照）
3. **测例软件观测**（围绕 git / vim / gcc / rustc 的安装、运行与结果验证）

**不做：** 主机通信、9p、前端、周期性后台采集、页表真实数据。

## 一、目录与文件布局

```
package/
├── osviz/                      # 日志基础设施
│   ├── make.sh
│   ├── include/osviz/log.h     # 测例程序 / 包装脚本调用的日志 API
│   ├── src/log.c
│   ├── osviz-snapshot          # 按需快照脚本（shell 或小型 C）
│   └── osviz-help              # 打印「现场赛该用哪些 Linux 命令看啥」
│
├── osviz-tests/                # 现场赛示例源码（可选）
│   ├── make.sh
│   ├── hello.c                 # gcc 测例
│   └── helloworld.rs           # rustc 测例
│
└── user-tests/                 # 用户自定义测例扩展（可选）
    ├── README.md
    └── my_test/
        └── main.c

运行时日志目录（Guest 内）：
/var/log/osviz/
├── events.jsonl                # 事件流，程序运行时追加
└── snapshots/                  # 按需生成，不自动刷
    └── snapshot-YYYYMMDD-HHMMSS.json
```

安装位置：

| 内容 | 路径 |
|------|------|
| git / vim / gcc / rustc | `/usr/bin/`（与官方测例镜像一致） |
| 测例源码产物 | `/root/` 或工作目录（如 `hello.c`、`helloworld.rs`、`a.out`） |
| 日志库头文件 | `/usr/include/osviz/log.h` |
| 按需快照工具 | `/usr/bin/osviz-snapshot` |
| 命令帮助 | `/usr/bin/osviz-help` |

## 二、日志系统设计

### 2.1 两类
Two kinds of logs

| 类型 | 文件 | 何时写 | 谁写 |
|------|------|--------|------|
| **事件日志** | `/var/log/osviz/events.jsonl` | 测例程序运行时，关键操作发生即写 | git / vim / gcc / rustc 包装脚本或 demo |
| **快照** | `/var/log/osviz/snapshots/snapshot-*.json` | **仅用户手动执行** `osviz-snapshot` | `osviz-snapshot` 脚本 |

**不做** 后台定时写 snapshot，避免大文件频繁生成。

### 2.2 事件日志格式（JSONL，一行一条）

```json
{"ts":1710000001.23,"module":"user","event":"git_commit","pid":100,"data":{"repo":"/root/proj","message":"add README.md"}}
```

字段约定：

- `ts`：Unix 时间戳（浮点）
- `module`：`boot` / `irq` / `mem` / `proc` / `fs` / `user`
- `event`：具体操作名
- `pid`：当前进程
- `data`：模块专有字段（可选）

测例程序在 git commit、gcc 编译、文件读写等点调用 `osviz_log_event()` 即可。

### 2.3 快照内容 — 对应课程 6 大模块

`osviz-snapshot` 一次生成一个 JSON，内含 6 个 section，**用 Linux 命令采集**：

| 模块 | section 名 | 采集方式（Guest 内已有命令） |
|------|-----------|------------------------------|
| 1. 启动 | `boot` | `dmesg \| head`；`/proc/cmdline`；`uname -a` |
| 2. 中断异常 | `irq` | `/proc/interrupts`；`/proc/softirqs` |
| 3. 内存管理 | `mem` | `free`；`/proc/meminfo`；可选指定 pid 的 `/proc/PID/maps` |
| 4. 进程管理 | `proc` | `ps aux` 或 `ps -eo pid,ppid,stat,cmd`；`/proc` 下 pid 列表 |
| 5. 文件系统 | `fs` | `df -h`；`mount`；`ls -li /` 或指定目录 |
| 6. 用户程序 | `user` | `git -v`；`gcc -v`；`rustc --version`；`vim --version`；`which git vim gcc rustc` |

**页表 section（仅占位，供后续前端可视化）：**

```json
"pagetable": {
  "status": "placeholder",
  "reason": "Linux 用户态无法直接遍历页表",
  "fallback": "/proc/PID/maps 可提供虚拟内存布局",
  "maps_sample": "..."
}
```

- 可附带某个 pid 的 `/proc/PID/maps` 文本作为 `maps_sample`
- **不** 尝试读 `/proc/PID/pagemap` 或写内核模块
- 前端以后只接这个字段结构，现在先把机制留好

## 三、用 Linux 命令「看」—— `osviz-help`

提供 `/usr/bin/osviz-help`，打印**现场赛测例**常用命令对照表，例如：

```text
=== 第一题：git ===
git help
git init
git add .
git commit -m "add README.md"
git log
git config --global user.name "youname"
git config --global user.email "alice@example.com"
git clone git@github.com:oscomp/xv6-riscv.git my-folder
git remote add me git@github.com:your/xv6-riscv.git
git push me
git pull me

=== 第二题：vim ===
vim -h
vim hello.c

=== 第三题：gcc ===
gcc --help
gcc hello.c && ./a.out

=== 第四题：rustc ===
rustc -h
rustc helloworld.rs && ./helloworld

=== 系统观测（辅助） ===
ps aux
free
cat /proc/meminfo
df -h
mount
cat /proc/interrupts

=== 按需快照 ===
osviz-snapshot
osviz-snapshot --pid 123    # 可选：附带某进程 maps
```

用户先在终端用这些命令完成现场赛测例，需要存档时再跑 `osviz-snapshot`。

## 四、日志库（给测例程序用）

### Step 1：实现 `package/osviz/`

- `log.h` + `log.c`：写 `/var/log/osviz/events.jsonl`
- API：
  - `osviz_log_init()` — 确保目录存在
  - `osviz_log_event(module, event, json_data)` — 追加一行
- 安装头文件和静态库（或单 `.c` 直接链入）

### Step 2：实现 `osviz-snapshot`

Shell 脚本即可（busybox `sh` + 已有工具）：

```bash
osviz-snapshot              # 生成全量 6 模块快照
osviz-snapshot --pid 123    # 额外带上该进程 maps
```

输出：`/var/log/osviz/snapshots/snapshot-20250526-143022.json`

### Step 3：实现 `osviz-help`

纯文本帮助，无逻辑。

### Step 4：安装 init 脚本（可选，很轻）

`S98osviz` 只做一件事：启动时 `mkdir -p /var/log/osviz/snapshots`。  
**不** 启动任何后台采集进程。

## 五、测例软件与示例源码

### 5.1 约定

- 现场赛二进制（git / vim / gcc / rustc）来自官方 Alpine ext4fs 镜像，装到 `/usr/bin/`
- 示例源码放：`package/osviz-tests/`（`hello.c`、`helloworld.rs`）
- 用户扩展放：`package/user-tests/<test_name>/`
- 鼓励在编译脚本或包装层 `#include <osviz/log.h>`

### 5.2 新增 `package/user-tests/make.sh`

逻辑：

1. 扫描 `package/user-tests/*/`（跳过无 `.c` / `.rs` 的目录）
2. 对每个 test 编译并复制到 `${TARGET_DIR}/root/` 或工作目录
3. 无 test 时不报错

### 5.3 官方示例 `package/osviz-tests/`

| 文件 | 对应现场赛小题 | 打的 event（示例） |
|------|--------------|-------------------|
| `hello.c` | 3.2 gcc hello.c && ./a.out | compile_start, compile_ok, run_output |
| `helloworld.rs` | 4.2 rustc helloworld.rs && ./helloworld | compile_start, compile_ok, run_output |

git / vim 可直接用系统命令验证，也可写轻量 shell 包装记录 event。

### 5.4 构建集成

在 `Makefile` 增加（顺序重要）：

```makefile
.PHONY: osviz
osviz:
	@./package/osviz/make.sh

.PHONY: user-tests
user-tests:
	@./package/user-tests/make.sh

.PHONY: osviz-tests
osviz-tests:
	@./package/osviz-tests/make.sh
```

在 `build.sh` 里，`rebuild-target-image` 之前插入：

```bash
make osviz || exit 1
make osviz-tests || exit 1    # 可选
make user-tests || exit 1
```

### 5.5 现场赛验证 checklist（写进 `package/user-tests/README.md`）

1. 确认 `git -v`、`gcc -v`、`rustc --version`、`vim --version` 与 `soft-info.txt` 一致
2. 构建：
   ```bash
   make osviz osviz-tests rebuild-target-image
   ```
3. QEMU 启动后进系统，按测例顺序验证：
   ```bash
   # Task 1.1
   git help

   # Task 1.2
   cd proj && git init
   cat >README.md
   git add .
   git commit -m "add README.md"
   git log

   # Task 2
   vim -h
   vim hello.c

   # Task 3
   gcc hello.c && ./a.out

   # Task 4
   rustc helloworld.rs && ./helloworld

   # 观测
   tail -f /var/log/osviz/events.jsonl
   osviz-snapshot
   cat /var/log/osviz/snapshots/snapshot-*.json
   ```

**增量构建**（只改了示例源码）：

```bash
make osviz-tests rebuild-target-image
```

## 六、典型使用流程（全在 Guest 终端）

```
# 1. 确认测例软件可用
git -v
gcc -v
rustc --version
vim --version

# 2. 跑现场赛测例
cd proj
git init
git add .
git commit -m "add README.md"
gcc hello.c && ./a.out
rustc helloworld.rs && ./helloworld

# 3. 实时看事件日志
tail -f /var/log/osviz/events.jsonl

# 4. 用 Linux 命令直接观察
git log
ps -eo pid,ppid,stat,cmd
df -h

# 5. 需要存档时，手动采快照（6 模块 + 页表占位）
osviz-snapshot
osviz-snapshot --pid 123

# 6. 查看快照
ls /var/log/osviz/snapshots/
cat /var/log/osviz/snapshots/snapshot-*.json | less

# 7. 不知道用什么命令时
osviz-help
```

一、总体思路 
通过从零开始实现一个简化版操作系统内核，深入理解操作系统底层原理，掌握
系统启动、内存管理、进程调度、中断处理、文件系统等核心机制，培养系统级编程
能力和工程实践能力。以程序加载 + 系统启动 + 内核管理 + 文件系统为核心，面向
“系统开发者”视角，用QEMU/Bochs 等现成模拟器即可运行。可在 QEMU/Bochs 启
动 操作系统并执行命令，实现基本内核功能。 
二、项目技术方案 
1. 系统启动（Bootloader） 
功能示例： 
• 实现从实模式到保护模式（x86）或从M 态到S 态（RISC-V）的切换 
• 加载内核镜像到内存正确位置并跳转执行 
• 初始化段描述符表（GDT/IDT）或中断向量表 
• 设置栈指针，为C 语言运行环境做准备 
• 输出启动日志信息，证明启动成功 
技术指标： 
• 能在模拟器中成功启动并进入内核主函数 
• 启动过程稳定可复现，无异常重启或死机 
2. 中断与异常处理 
功能示例： 
• 实现中断描述符表（IDT）的初始化与注册 
• 实现时钟中断处理程序，支持定时功能 
• 实现键盘中断处理，支持基本输入 


• 实现系统调用接口（int 0x80 或 ecall），支持用户态到内核态切换 
• 实现缺页异常处理 
技术指标： 
• 时钟中断频率稳定（如100Hz） 
• 键盘中断能正确读取扫描码并转换为ASCII 字符 
• 系统调用支持：打印、进程创建、进程退出、文件读写 
3. 内存管理 
功能示例： 
• 实现物理内存的探测与管理（可用内存区域识别） 
• 实现物理页框分配器（基于位图或链表的伙伴算法/简单分配器） 
• 实现虚拟内存管理：页表创建、映射、解除映射 
• 实现内核堆内存分配（kmalloc/kfree） 
• 实现用户态内存空间布局（代码段、数据段、堆、栈） 
技术指标： 
• 支持至少 4MB 物理内存管理 
• 页大小为 4KB 
• 支持按需分页（Demand Paging） 
• 内存分配无泄漏，支持释放后重用 
4. 进程管理 
功能示例： 
• 实现进程控制块（PCB）数据结构 
• 实现进程创建（fork/exec） 
• 实现进程调度器（支持至少两种算法：FCFS 和 RR） 
• 实现进程状态转换：就绪、运行、阻塞、僵尸 
• 实现进程间同步机制：信号量、互斥锁 
• 实现进程退出与等待（wait/waitpid） 
技术指标： 


• 支持多个并发进程 
• 时间片轮转调度时间片可配置（默认10ms） 
• 进程切换开销小于 1ms（在 QEMU 中） 
• 支持父子进程关系树 
5. 文件系统 
功能示例： 
• 实现简化版文件系统（如基于内存的 RAMFS 或简化 EXT2） 
• 支持文件/目录的创建、删除、打开、关闭、读写 
• 实现文件描述符表管理 
• 实现路径解析（绝对路径与相对路径） 
• 支持标准输入输出重定向 
技术指标： 
• 支持至少 128 个文件，单文件最大 64KB 
• 目录层级至少支持 3 层嵌套 
• 文件读写支持 seek 操作 
• 提供 mkfs 工具初始化文件系统镜像 
6. 用户程序加载与执行 
功能示例： 
• 实现 ELF 格式可执行文件的解析与加载 
• 实现用户态栈初始化（参数传递、环境变量） 
• 实现系统调用封装库（libc 简化版） 
• 实现 shell 命令行解释器，支持基本命令： 
o ls：列出目录内容 
o cat：显示文件内容 
o echo：输出字符串 
o ps：显示进程列表 
o kill：发送信号终止进程 


o exec：执行外部程序 
技术指标： 
• 至少能加载运行 5 个不同的用户程序 
• shell 支持命令解析、参数传递、管道 
• 用户程序崩溃不影响内核稳定性
